# 要求定義・要件定義 — iterm2-cli

作成日: 2026-06-15

iTerm2 をスクリプト/エージェントから操作する CLI の、作る理由・ユースケース・機能/非機能要件・受け入れ基準を定義する。設計の詳細は [design.md](./design.md)、調査根拠は [research.md](./research.md) を参照。

---

## 1. 背景と目的（なぜ作るか）

iTerm2 には既に同梱 CLI `it2api` があり、Python API のほぼ全操作（列挙・送信・分割・画面取得・変数 等）を CLI 化している。したがって本 CLI は **「it2api で満たせないギャップ」に価値を集中**させない限り存在意義が薄い。

最有力ユースケースは **orchestrator**（複数 Claude ペインのオーケストレーション）。現状 オーケストレータ は `オーケストレータ.py` / `(利用側スクリプト)` 等で iTerm2 制御をアドホック実装しており、これを **再利用可能な制御基盤（ライブラリ＋CLI）へ一本化**することが本 CLI の目的。

---

## 2. ユースケース（要求定義）

| UC | 主体 | シナリオ |
|---|---|---|
| UC1 | オーケストレータ（人/AI） | 新しいペイン（タブ/分割）を作り、そこで Claude Code 等のエージェントを起動する |
| UC2 | オーケストレータ | 名前/ラベルで識別したペインに、後からプロンプトを送信する（本文送信→確定キー） |
| UC3 | オーケストレータ | 対象ペインが処理中（busy）か・入力待ちか・完了したかを判定し、完了を待って次を送る |
| UC4 | オーケストレータ | 対象ペインの画面内容を読み取り、結果を機械可読（JSON）で取得する |
| UC5 | エージェント（各ペイン） | 自分の状態（実行中/要入力/完了/進捗）を UI と上位に報告する |
| UC6 | オーケストレータ | フォーカス移動・ペイン/タブのクローズなどライフサイクル操作を行う |
| UC7 | スクリプト | セッション/ペイン階層を一覧し、ID やプロパティを機械可読で取得する |
| UC8 | 高頻度バッチ | 多数の送信/読取を低レイテンシで連続実行する |

## 3. it2api ギャップ分析（差別化の根拠）

| # | ギャップ | it2api の現状 | 本 CLI が足す価値 | 関連 UC |
|---|---|---|---|---|
| G1 | 永続的なペイン識別（ラベル・再起動跨ぎ） | session_id を毎回 `list-sessions` で探す。命名/管理なし | session_id↔label の最小マッピングと解決層 | UC2,UC7 |
| G2 | 高レベル操作（完了待ち・busy 検知・リトライ） | 単発操作のみ。busy 検知なし | `wait`/`busy`、OSC・hook 駆動の完了検知 | UC3 |
| G3 | 構造化出力（JSON） | 人間向けテキスト中心 | `--json` を全コマンドで標準化 | UC4,UC7 |
| G4 | 起動レイテンシ | コマンド毎に websocket 接続+認証 | 任意のデーモン化（同一コマンド表面） | UC8 |
| G5 | 依存導入の手間 | `iterm2` pip を各自導入が必要 | `uv`（PEP 723）で自己完結 | 全般 |
| G6 | 送信作法（bracket-paste・本文と確定キー分離・遅延） | 生 send-text のみ | send/send-key 分離＋送信作法を内蔵 | UC2 |
| G7 | 状態報告プリミティブ | なし | `set-status`/`set-progress` を UI（名前/バッジ/変数）へ反映 | UC5 |

**判断**: G1/G2/G3/G6/G7 は it2api では埋まらず、オーケストレータ が現にアドホック実装している。**新 CLI を作る価値あり。**

**it2api との関係（完全代替方針）**: it2api 自体が `iterm2` Python API の薄いラッパに過ぎず、特権的機能を一切持たない。本 CLI も同じ Python API を使うため、it2api でできることは全て自前実装できる。むしろ it2api にシェルアウトすると「プロセス毎起動・テキストのみ出力・接続を共有できずデーモン化の恩恵が消える」ため劣る。したがって:
- **it2api への実行時依存をゼロにする**（シェルアウトしない）。全操作を Python API へ直接実装する。← 設計判断として即確定。
- **機能網羅で it2api を完全代替する**ことを目標に置く。ただし全操作（profiles/arrangements/tmux/monitors 等）の網羅はスコープが広いため段階的に進める（ギャップ G1〜G7 を優先、残りは順次移植）。
- it2api は**アンインストールしない／できない**（iTerm2.app 同梱）。**移植元のリファレンス**として残し、依存はしない。

## 4. 機能要件（FR）

| FR | 要件 | 受け入れ基準 |
|---|---|---|
| FR1 | セッション/ペイン階層の列挙 | `list --json` が window/tab/session の階層と session_id を返す |
| FR2 | テキスト送信（本文）と確定キーの分離 | `send <target> <text>` は本文のみ送出、`send-key <target> enter` で確定。bracket-paste 安全 |
| FR3 | 画面内容読取 | `read <target> [--tail N] [--json]` が現在の画面行を返す |
| FR4 | busy/完了判定 | `busy <target>` が exit code/JSON で状態を返す。検知は OSC9/99/777・hook 駆動を第一、画面マーカーをフォールバック |
| FR5 | 完了待ち | `wait <target> [--timeout]` が完了まで（またはタイムアウトまで）ブロックする |
| FR6 | ペイン分割・タブ/ウィンドウ作成 | `split`/`tab` が新規セッションを作り session_id を返す |
| FR7 | フォーカス・クローズ | `focus`/`close <target>` が対象に作用 |
| FR8 | 変数 get/set | `var get|set <target> <name> [value]` |
| FR9 | 状態報告 | `set-status`/`set-progress` がセッション名/バッジ/変数に反映 |
| FR10 | 通知 | `notify [--title --body]` |
| FR11 | `<target>` 解決 | `--session <id>` → ラベル → `$ITERM_SESSION_ID`（current）の順。省略時は current |
| FR12 | デーモン | `daemon` 起動時は Unix socket 経由で低レイテンシ。未起動時は都度接続にフォールバック（同一コマンド表面） |

## 5. 非機能要件（NFR）

| NFR | 要件 |
|---|---|
| NFR1（言語） | Python 実装。iTerm2 の実用 API が Python のみ、かつ オーケストレータ 既存資産も Python のため（[design.md](./design.md) の言語比較参照） |
| NFR2（依存） | `uv`（PEP 723 インラインメタデータ）で自己完結。グローバル pip 汚染を避ける |
| NFR3（レイテンシ） | 都度接続で実用範囲、高頻度用途はデーモンで低レイテンシ。同一コマンド表面で両立 |
| NFR4（出力） | 全コマンドで `--json` を提供。人間向け既定 + 機械可読オプション |
| NFR5（再利用） | コア機能はライブラリ層（`iterm2_cli` パッケージ）として オーケストレータ から import 可能 |
| NFR6（互換） | iTerm2 3.0+（Python API）。API 許可は初回ダイアログ or `ITERM2_COOKIE` |
| NFR7（堅牢性） | 接続失敗・認証失敗・対象セッション消失を明示エラーで返す（silent fail しない） |
| NFR8（安全） | 不可逆操作（リポジトリ push/名前変更、credential、API 許可設定の改変）は実行せず停止（オーケストレータ 不可逆操作 規約） |

---

## 6. スコープ外（今回は作らない）

- it2api 全操作（profile/arrangement/tmux 連携/各種 monitor 等）の**網羅的カバーは本フェーズ対象外**（完全代替は段階的目標。本フェーズはギャップ G1〜G7 と中核操作を優先）。なお「it2api にシェルアウトしない＝Python API 直接実装」は対象内の設計判断。
- iTerm2 以外のターミナルエミュレータ対応（将来の抽象化候補だが本フェーズ対象外）。
- ブラウザ自動化（cmux にはあるが本 CLI のコア責務外）。

---

## 7. 受け入れ基準（要件定義フェーズの完了条件）

本フェーズ（要件定義＋設計）の成果物は以下を満たすこと:
1. it2api ギャップ分析と完全代替方針（Python API 直接実装・実行時依存ゼロ・段階的網羅）が明示されている（本書 §3）。
2. 言語が根拠付きで Python に確定している（[design.md](./design.md)）。
3. レイテンシ対策（都度接続＋任意デーモン）の方針が定まっている。
4. 全コマンド × 引数 × socket method の第一仕様が示されている（[design.md](./design.md)）。
5. オーケストレータ との統合方針（ライブラリ層・デーモン実体）が定まっている。
